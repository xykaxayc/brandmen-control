# Brandmen: логика работы системы

Актуально для ветки `main`, версия `v0.92.0`.

> Важно: на момент составления документа боевой компьютер
> `DESKTOP-UOSQOOT` работал на `v0.90.0`. Поэтому новая логика автоматического
> применения плейлиста после синхронизации на нём ещё не выполнялась.

## 1. Назначение системы

Brandmen управляет Android-планшетами в барбершопах. Планшеты показывают
рекламные ролики, а Windows/macOS-приложение позволяет сотруднику:

- видеть, какие планшеты находятся в сети и что на них играет;
- добавлять, удалять и упорядочивать ролики;
- синхронизировать контент на один или все планшеты;
- запускать воспроизведение;
- управлять громкостью, яркостью и экраном;
- обновлять Android-плеер;
- получать диагностические логи.

Интерфейс рассчитан на пользователя без технических знаний.

## 2. Компоненты

### 2.1. Пульт

Flutter-приложение для Windows и macOS.

Основные задачи:

- хранить список планшетов;
- хранить папку с роликами и порядок воспроизведения;
- конвертировать видео в совместимый формат;
- отправлять ролики и команды по локальной сети;
- использовать ADB как резервный канал;
- показывать статусы, ошибки и скриншоты;
- проверять собственные обновления;
- отправлять диагностические логи.

### 2.2. Android-плеер

Нативное Android-приложение `com.brandmen.ads`.

Основные задачи:

- проигрывать локальный плейлист по кругу;
- принимать ролики и команды через HTTP на порту `5011`;
- сообщать состояние воспроизведения;
- работать в foreground service;
- самостоятельно восстанавливать воспроизведение и Wi-Fi;
- устанавливать обновления APK;
- применять политики Device Owner.

### 2.3. Каналы связи

- HTTP `5011` на планшете — основной канал управления и синхронизации.
- ADB over Wi-Fi `5555` — резервный канал.
- HTTP `5010` на компьютере — регистрация планшетов и выдача файлов.
- mDNS и UDP broadcast — обнаружение компьютера в локальной сети.
- Внешний сервер логов — диагностика Windows-приложения.
- Outbound command channel — резервный канал удалённых команд для планшета.

## 3. Запуск Windows-приложения

При старте приложение:

1. Загружает настройки.
2. Загружает выбранный бренд-пакет.
3. Определяет папку с роликами.
4. Запускает локальный HTTP-сервер на порту `5010`.
5. Запускает mDNS и UDP-обнаружение.
6. Загружает сохранённый список планшетов.
7. Проверяет состояние каждого планшета.
8. Через пять секунд проверяет обновление программы.
9. Далее проверяет обновления каждые 30 минут.
10. Через 45 секунд отправляет диагностический лог.
11. Далее отправляет полный лог каждые 15 минут.
12. Новые строки логов пытается отправлять каждые три секунды.

## 4. Добавление планшета

### 4.1. Сопряжение по Wi-Fi

1. Пользователь включает режим сопряжения на компьютере.
2. Режим действует около 30 секунд.
3. Планшет ищет компьютер:
   - по сохранённому IP;
   - через mDNS;
   - через UDP broadcast.
4. Планшет отправляет запрос:
   `POST http://<IP-компьютера>:5010/api/register`.
5. Компьютер сохраняет IP и название планшета.
6. Режим сопряжения отключается.
7. Пульт пробует подключить ADB по сети.

Обычное Wi-Fi-сопряжение не назначает Device Owner.

### 4.2. Полная подготовка через USB

Для полноценного управления устройство должно быть чистым, без добавленных
аккаунтов.

1. Планшет подключается по USB.
2. Пульт обнаруживает его через ADB.
3. Определяет IP планшета.
4. Выполняет:
   `dpm set-device-owner com.brandmen.ads/.DeviceAdminReceiver`.
5. Проверяет реальный статус Device Owner.
6. Только после подтверждения включает `adb tcpip 5555`.
7. Подключается к планшету через Wi-Fi.
8. Сохраняет планшет в списке.

Если на устройстве уже существуют аккаунты, Android запрещает назначение
Device Owner. В этом случае нужен сброс устройства или удаление аккаунтов.

## 5. Определение состояния планшета

Для каждого сохранённого IP пульт проверяет:

1. Доступен ли HTTP-плеер на `:5011`.
2. Подключён ли ADB на `:5555`.
3. Если ADB не подключён — пытается переподключить его.
4. Если HTTP доступен — запрашивает `/health`.

Из `/health` приложение получает:

- версию Android-плеера;
- свободное и общее место;
- текущий ролик;
- индекс ролика;
- количество роликов;
- признак `playing`;
- заряд батареи;
- статус Device Owner;
- подпись APK;
- разрешение установки APK;
- исключение из оптимизации батареи;
- разрешение показа поверх других окон.

Планшет считается онлайн, если доступен HTTP или ADB.

Полноценный статус обновляется при открытии экрана и ручном обновлении.
Скриншоты онлайн-планшетов обновляются примерно каждые пять минут.

## 6. Работа с роликами

### 6.1. Папка-источник

По умолчанию используется `Документы/BrandmenVideos`. Пользователь может
выбрать другую папку.

Поддерживаемые расширения:

- `.mp4`;
- `.mov`;
- `.mkv`;
- `.avi`;
- `.webm`.

### 6.2. Порядок воспроизведения

Порядок хранится в двух файлах:

- `.brandmen_order.json` — внутренний порядок пульта;
- `playlist.m3u` — плейлист Android-плеера.

При добавлении, удалении или перетаскивании роликов `playlist.m3u`
перезаписывается.

## 7. Конвертация видео

Перед синхронизацией вся папка проверяется и приводится к единому профилю.

Целевой формат:

- контейнер MP4;
- H.264 Baseline;
- H.264 Level 3.1;
- разрешение не больше `1280×720`;
- `yuv420p`;
- без B-кадров;
- постоянные 30 FPS;
- AAC, 48 кГц, стерео;
- `faststart`;
- максимальный видеобитрейт около 5 Мбит/с.

Алгоритм:

1. `.mov`, `.mkv`, `.avi` и `.webm` конвертируются в отдельный `.mp4`.
2. Существующие `.mp4` проверяются через `ffprobe`.
3. Несовместимый MP4 перекодируется на месте.
4. Исходники других форматов не удаляются.
5. Для подготовленного файла создаётся служебный marker версии профиля.
6. Плейлист переписывается так, чтобы ссылаться на MP4.

Если отсутствует `ffmpeg` или хотя бы один файл не удалось подготовить,
синхронизация полностью останавливается. Предыдущий рабочий набор на планшетах
не заменяется частично.

## 8. Синхронизация одного планшета

Текущая последовательность:

1. Проверить и конвертировать все ролики.
2. Попытаться использовать HTTP `5011`.
3. Если HTTP недоступен или завершился ошибкой — использовать ADB.
4. Попытаться передать актуальный бренд-пакет.
5. Получить список файлов планшета.
6. Проверить свободное место.
7. Сравнить локальные и удалённые файлы.
8. Передать отсутствующие или отличающиеся ролики.
9. Передать `playlist.m3u` последним.
10. После успешной загрузки удалить лишние ролики.
11. Отправить `/api/control/launch`.
12. Android перечитывает плейлист и начинает с первого ролика.
13. Windows до 30 секунд проверяет `/api/control/now`.
14. Операция считается успешной только после `playing=true`.

Шаги 11–14 добавлены в `v0.92.0`. Раньше передача `playlist.m3u` не заставляла
уже работающий Android-плеер перечитать плейлист. Контент появлялся только
после ручного нажатия «Обновить» на планшете.

## 9. Массовая синхронизация

1. Берутся планшеты, помеченные онлайн на момент запуска.
2. Конвертация выполняется один раз для общей папки.
3. Файлы передаются на планшеты последовательно, чтобы не перегружать Wi-Fi.
4. Успешно синхронизированные планшеты добавляются в очередь применения.
5. Команда запуска отправляется группами по три планшета.
6. Каждый планшет отдельно подтверждает `playing=true`.
7. Ошибка одного планшета не останавливает остальные.

Офлайн-планшеты пропускаются. Очереди автоматического применения контента после
их возвращения в сеть сейчас нет.

## 10. Логика Android-плеера

Android-приложение:

1. Запускает foreground service.
2. Поднимает HTTP-сервер на порту `5011`.
3. Удерживает Wi-Fi lock.
4. Загружает `/sdcard/Movies/ads/playlist.m3u`.
5. Проигрывает ролики по кругу.
6. После завершения переходит к следующему ролику.
7. Если ролик выдаёт ошибку — пропускает его.
8. Если плейлиста нет или он пустой — сортирует все видео по имени.
9. Если видео нет — показывает брендированную заставку.
10. По команде `launch` перечитывает плейлист и начинает с первого ролика.

### 10.1. Самовосстановление воспроизведения

Раз в 20 секунд плеер:

- запускает воспроизведение снова, если видео не играет;
- переходит к следующему ролику, если позиция не меняется две проверки подряд;
- после необработанного падения планирует перезапуск примерно через 1,5 секунды.

### 10.2. Самовосстановление сети

Раз в минуту:

1. Проверяется наличие локальной сети.
2. Если Wi-Fi отключён — приложение пытается его включить.
3. Выполняется переподключение к сохранённой сети.
4. Если сети нет 10 минут и приложение является Device Owner — планшет
   перезагружается.
5. Повторный автоматический ребут разрешён не чаще одного раза в час.
6. В 05:00 выполняется плановый ежедневный ребут Device Owner-планшета.

Дополнительный watchdog запускается примерно каждые 15 минут, даже если
основной сервис был остановлен системой.

## 11. Device Owner

Device Owner даёт:

- восстановление Wi-Fi;
- удалённую перезагрузку;
- автоматическую выдачу разрешений;
- более надёжный запуск после перезагрузки;
- тихую установку APK;
- удержание экрана включённым при зарядке.

Жёсткий kiosk mode намеренно не используется. Планшет можно свернуть, закрыть
и обслуживать вручную.

Device Admin и Device Owner — разные статусы. Наличие Device Admin ещё не
означает полноценного управления.

## 12. Обновления

### 12.1. Windows/macOS

1. Приложение проверяет GitHub Releases при запуске.
2. Начиная с `v0.91.0`, повторяет проверку каждые 30 минут.
3. Ищет самый новый релиз с ZIP для текущей платформы.
4. Показывает предложение обновиться.
5. После подтверждения скачивает и применяет обновление.

Обновление не принудительное: пользователь должен подтвердить установку.

### 12.2. Android

APK передаётся:

1. по HTTP;
2. через ADB;
3. через системное окно установки как ручной fallback.

Device Owner позволяет выполнить тихую установку. На обычном планшете человек
должен подтвердить установку на экране.

## 13. Логи

Windows-приложение:

- пишет локальный лог;
- отправляет полный лог примерно каждые 15 минут;
- отправляет новые строки почти в реальном времени;
- повторно ставит строки в очередь, если live-отправка не удалась.

В лог записываются:

- действия пользователя;
- доступность HTTP и ADB;
- версии планшетов;
- текущий ролик и состояние `playing`;
- передача и удаление файлов;
- запуск и проверка воспроизведения;
- обновления приложений;
- ошибки Device Owner;
- ошибки конвертации и сети.

Android предоставляет последние строки `logcat` через `/log`.

## 14. Известные слабые места

### 14.1. Синхронизация контента

1. Файлы сравниваются преимущественно по имени и размеру, а не по SHA-256.
   Новое содержимое с тем же размером может быть ошибочно признано актуальным.
2. Нет единого `content_revision` или `deployment_id`.
3. Android принимает файл во временный `.part`, но не проверяет, что получил
   весь заявленный `Content-Length`, прежде чем заменить рабочий файл.
4. Нет проверки checksum после загрузки.
5. Весь набор не переключается атомарно. Атомарно заменяется отдельный файл,
   а плейлист передаётся последним.
6. ADB-ветка может удалять лишние файлы даже после ошибки загрузки. В
   HTTP-ветке удаление при ошибке правильно блокируется.
7. `playing=true` доказывает только воспроизведение какого-либо ролика, но не
   доказывает применение именно нового плейлиста.
8. Нет автоматического отката на предыдущий рабочий набор.

### 14.2. Состояние флота

1. Офлайн-планшет пропускается и не получает контент автоматически после
   возвращения в сеть.
2. Список хранится по IP. После изменения DHCP-адреса планшет может считаться
   потерянным.
3. Нет постоянного короткого фонового опроса всех статусов.
4. Старые плееры возвращают `404` для новых команд, но строгой матрицы
   совместимости функций и версий нет.
5. Пустой или повреждённый плейлист маскируется воспроизведением файлов по
   алфавиту.

### 14.3. Безопасность

1. HTTP API планшета в локальной сети не имеет полноценной индивидуальной
   аутентификации.
2. Любой клиент в той же сети потенциально может отправлять команды и файлы.
3. Windows-клиент разрешает неподтверждённый TLS-сертификат для GitHub-доменов,
   что помогает при антивирусном прокси, но снижает безопасность обновления.

## 15. Рекомендуемая целевая логика

### 15.1. Формирование deployment

Windows создаёт неизменяемый пакет:

- `deployment_id`;
- версия бренд-пакета;
- упорядоченный плейлист;
- SHA-256 каждого файла;
- размер и длительность;
- параметры кодека;
- дата создания.

Весь пакет сначала валидируется локально.

### 15.2. Двухфазная синхронизация

1. Windows отправляет `prepare deployment`.
2. Планшет создаёт отдельную staging-папку.
3. Файлы загружаются с возможностью продолжения.
4. Каждый файл проверяется по SHA-256.
5. После получения всех файлов планшет отвечает `ready_to_commit`.
6. Windows отправляет `commit`.
7. Планшет атомарно переключает активную папку и плейлист.
8. Плеер запускает новый набор.
9. Планшет возвращает:
   - `active_deployment_id`;
   - `playlist_hash`;
   - текущий ролик;
   - `playing=true`.
10. Если запуск не подтвердился — выполняется откат.

### 15.3. Желаемое состояние планшета

Для каждого устройства сервер хранит:

- постоянный `device_id`;
- текущий IP;
- `desired_deployment_id`;
- `active_deployment_id`;
- версию плеера;
- время последней связи;
- результат последней синхронизации.

Офлайн-планшет после появления в сети сам получает нужный deployment.

### 15.4. Машина состояний

Нормальный сценарий:

`offline → online → preparing → uploading → verifying → activating → playing`

Сценарий ошибки:

`uploading → failed → rollback → playing_previous`

Оператор должен видеть текущее состояние простыми словами:

- «Не в сети»;
- «Подготавливаю ролики»;
- «Передаю 4 из 12»;
- «Проверяю»;
- «Показывает новый плейлист»;
- «Ошибка, оставлен предыдущий плейлист».

### 15.5. Дополнительные требования

- Использовать `device_id`, а не IP, как главный идентификатор.
- Добавить индивидуальный токен каждого планшета.
- Проверять совместимость версии плеера до синхронизации.
- Хранить предыдущий рабочий deployment для отката.
- Разделить состояния «файл передан», «плейлист применён» и «видео играет».
- Автоматически обновлять IP через discovery.
- Добавить фоновый опрос статусов с разумным интервалом.

## 16. Задача для дополнительного анализа

Этот документ можно передать другой нейросети со следующим запросом:

> Проанализируй текущую архитектуру Brandmen. Предложи надёжную state machine
> синхронизации рекламного контента для парка Android-планшетов в нестабильной
> локальной Wi-Fi-сети. Требования: атомарное применение набора роликов,
> SHA-256-проверка, возобновляемая загрузка, автоматическая доставка после
> возвращения планшета в сеть, подтверждение активной версии плейлиста, откат
> на предыдущий рабочий deployment, поддержка старых планшетов и простой UX
> для не технического пользователя. Отдельно найди логические, сетевые и
> security-проблемы в текущем процессе и предложи поэтапный план миграции без
> остановки работающего флота.

## 17. Ключевые файлы

- `lib/main.dart` — пользовательские сценарии и массовая синхронизация.
- `lib/adb_manager.dart` — статусы, ADB, HTTP fallback и передача файлов.
- `lib/device_http.dart` — HTTP-клиент планшета.
- `lib/transcoder.dart` — проверка и конвертация видео.
- `lib/server.dart` — локальный сервер компьютера и регистрация.
- `lib/updater.dart` — обновление пульта и получение APK.
- `android-player/src/main/java/com/brandmen/ads/MainActivity.java` —
  воспроизведение и интерфейс планшета.
- `android-player/src/main/java/com/brandmen/ads/MediaServer.java` —
  HTTP API планшета.
- `android-player/src/main/java/com/brandmen/ads/PlayerService.java` —
  foreground service.
